# 前端 Top 检查更新报告

## 1. 检查口径

本报告只记录 `top/front_end` 相对新版 `simulator-new/front-end` 的检查结果。当前前端包保留两层视角：

本次前端包已按老板/学长最新要求从“当前默认有效分支”改为“全分支训练口径”：以 `SimCpu::front_cycle()` 作为函数运行顺序标准，同时展开 `CONFIG_BPU` 主路径、Oracle 参考路径、2-Ahead/NLP、ICache slot1、fetch-to-ICache bypass、ICache-to-predecode bypass 和 front2back-output bypass。

| 层级 | 当前处理 |
|---|---|
| RTL wrapper 层 | `front_top.v` 仍按 `bpu_top / fetch_address_fifo_top / icache_top / predecode_top / instruction_fifo_top / ptab_top / predecode_checker_top / front2back_fifo_top` 8 个一级 wrapper 组织，格式与 `back_end` 一致。 |
| comb 训练单元层 | 交互图和说明按全分支训练口径展开。截图中前 4 个 `bpu_hist_*` 是 `bpu_hist_comb` 的 generated 子块；第 5 个 `bpu_post_read_req_comb` 不属于 hist；`bpu_nlp_comb` 已重新纳入 2-Ahead 分支。 |

| 项目 | 更新位置 | 说明 |

## 3. 当前仍需继续推进的项

| 项目 | 当前处理 | 原因 |
|---|---|---|
| ICache 到 predecode/checker 的 bypass | 已在 HTML 和 `.v` 中展开 | 后续需要根据真实 FIFO wrapper 增加正式可综合旁路 mux/时序寄存器验证。 |
| ICache slot1 双请求返回路径 | 已在 HTML 和 `.v` 中展开 | 当前 `.v` 先暴露 ICache slot1 输出；要完全等价 C++ 仍需 fetch-address FIFO / instruction FIFO 支持双读双写或顺序两次 comb 的 RTL 表达。 |
| `ENABLE_2AHEAD` 第二次 fetch-address 写入 | 已在 HTML 和 `.v` 中展开 | 当前 `.v` 以参数和显式线网保留分支；后续需要确认 FIFO 多写端口和 PTAB dummy/mini-flush 的可综合实现。 |
| ICache PTW / AXI memory runtime 端口 | 保留在 ICache 内部，不抬到 `front_top.v` | `front_module.h` 中是运行时绑定接口；是否作为前端 top 对外端口需要后续统一 SoC/AXI 边界后再定。 |

## 4. HTML 检查结论

`front_top_interactive.html` 当前表达的是 comb 训练单元层，不是 8 个 wrapper 层。检查结果：

| 检查项 | 结果 |
|---|---|
| `<script>` 语法 | 通过 `new Function(script)` 静态解析 |
| 正式 comb 单元数 | 27 + `bpu_nlp_comb` 全分支项 |
| `bpu_hist_comb` generated 子块 | 4 个 |
| 连线数 | 58 |
| 路径数 | 11 |
| 缺失模块引用 | 0 |
| 缺失连线依据 | 0 |
| 缺失路径依据 | 0 |
| 源码行号有效性 | 已检查，所有 `simulator-new/...:line` 均能落到真实文件和行号 |

结论：HTML 可以继续作为汇报用交互式 Top 图。图二前 4 个 `bpu_hist_*` 子块建议只作为 `bpu_hist_comb` 的展开显示；2-Ahead/NLP、slot1 和 bypass 则作为全分支训练路径单独展开。
