# 前端 Top 检查更新报告

## 1. 检查口径

本报告只记录 `top/front_end` 相对新版 `simulator-new/front-end` 的检查结果。当前前端包保留两层视角：

本次前端包按老板/学长最新要求回到 **27 个正式 comb 训练边界口径**：以 `SimCpu::front_cycle()` 作为函数运行顺序标准，HTML 只把 27 个正式 comb 作为模块框展示；helper、内部子函数和默认关闭分支只保留在源码依据/说明中，不再单独展开成模块。

| 层级 | 当前处理 |
|---|---|
| RTL wrapper 层 | `front_top.v` 仍按 `bpu_top / fetch_address_fifo_top / icache_top / predecode_top / instruction_fifo_top / ptab_top / predecode_checker_top / front2back_fifo_top` 8 个一级 wrapper 组织，格式与 `back_end` 一致。 |
| comb 训练单元层 | 交互图和说明按“清单 comb 训练边界”计算：只展示 27 个已有位宽的正式 comb；`bpu_nlp_comb`、Oracle、slot1、bypass 与 `bpu_hist_*` 生成子函数均不作为单独模块框。截图中前 4 个 `bpu_hist_*` 属于 `bpu_hist_comb` 内部/生成子函数；第 5 个 `bpu_post_read_req_comb` 不属于 hist。 |

## 3. 当前仍需继续推进的项

| 项目 | 当前处理 | 原因 |
|---|---|---|
| ICache 到 predecode/checker 的 bypass | 默认关闭，不进入 27 模块图 | 若后续打开，需要根据真实 FIFO wrapper 增加正式可综合旁路 mux/时序寄存器验证。 |
| ICache slot1 双请求返回路径 | 默认关闭，不进入 27 模块图 | 当前 true ICache 默认不走 slot1；若切到 ideal dual-request，需要确认 fetch-address FIFO / instruction FIFO 支持双读双写或顺序两次 comb 的 RTL 表达。 |
| `ENABLE_2AHEAD` 第二次 fetch-address 写入 | 默认关闭，不进入 27 模块图 | 当前默认定义 `FRONTEND_DISABLE_2AHEAD`；若打开，需要确认 FIFO 多写端口和 PTAB dummy/mini-flush 的可综合实现。 |
| ICache PTW / AXI memory runtime 端口 | 保留在 ICache 内部，不抬到 `front_top.v` | `front_module.h` 中是运行时绑定接口；是否作为前端 top 对外端口需要后续统一 SoC/AXI 边界后再定。 |

## 4. HTML 检查结论

`front_top_interactive.html` 当前表达的是 comb 训练单元层，不是 8 个 wrapper 层。检查结果：

| 检查项 | 结果 |
|---|---|
| `<script>` 语法 | 通过 `new Function(script)` 静态解析 |
| 正式 comb 单元数 | 27 个 |
| `bpu_hist_comb` generated 子块 | 不单独展开；只在 `bpu_hist_comb` 的源码依据中说明 |
| 连线数 | 39 |
| 路径数 | 6 |
| 缺失模块引用 | 0 |
| 缺失连线依据 | 0 |
| 缺失路径依据 | 0 |
| 源码行号有效性 | 已检查，所有 `simulator-new/...:line` 均能落到真实文件和行号 |

结论：HTML 可以继续作为汇报用交互式 Top 图。图二前 4 个 `bpu_hist_*` 子函数只作为 `bpu_hist_comb` 的源码依据，不再单独展开；其它 helper 型 `*_comb` 依据 `FRONTEND_COMB_SEQ_WRITING_STANDARD.md` 的“清单 comb vs 非清单 comb”口径，不进入正式统计。2-Ahead/NLP、slot1 和 bypass 当前不作为 27 个正式模块展示。
